## Findings considered and rejected

- **Pagination on list queries** (getUsers/getEvaluations/getManuals/etc., no
  `take`): real but not worth doing at thesis scale (handfuls of rows). The
  data model won't reach a size where full-table fetches hurt before the
  defense. Revisit only if the app goes to real production volume.
- **Migrate all data pages to Server Components** (the documented "Server
  Components fetch data" convention is violated repo-wide; every page is
  `"use client"` + TanStack Query calling server actions): a real architectural
  drift, but effort L, risk MED, and it changes the auth/session flow on every
  page. High-churn, low incremental value for a working thesis app — not worth
  the regression risk now. Documented here so it isn't re-audited; if pursued,
  migrate ONE route as a reference first.
- **`lucide-react@^1.18.0` "unusual major"** (DEPS smell): rejected — the
  lockfile resolves it to a real `lucide-react@1.18.0(react@19.2.4)`. Not a
  fork, not a problem. False alarm.
- **`middleware.ts` for route-level auth**: folded into plan 001's Maintenance
  notes as optional defense-in-depth rather than its own plan — the Server
  Action guards are the real boundary and cover the exposure.
- **Extracting shared StatusBadge / confirm-delete-table** (DEBT-03,
  triplicated UI patterns): real duplication but low harm and low leverage
  (each copy is ~20 lines, stable). Not planned; pick it up opportunistically
  when next touching those components.
- **Magic-byte upload validation**: extension + MIME + size (plan 004) is
  proportionate for an internal tool. Deferred, noted in plan 004.
